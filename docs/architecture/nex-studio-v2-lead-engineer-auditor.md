# NEX Studio v2.0.0 — "Lead Engineer + Auditor" — Design

> Living design doc, built through the Director↔Dedo consultation (started 2026-06-25).
> **Process:** Director asks about a part → Dedo consults/answers → the final decision is recorded here.
> Supersedes the 5-role serial waterfall (Designer / Customer / Implementer / Auditor / Coordinator).

## Core architecture (one line)
A **Lead Engineer** (SK UI: **Hlavný vývojár**) — one strong senior agent that does the core work itself and dynamically spawns ephemeral helper agents for sub-tasks, holding full context end-to-end — plus an independent **Auditor** (the verifier: release-acceptance + adversarial spot-checks) + a configurable **autonomy dial**. Replaces the 5-role serial pipeline. Rationale: dogfood metrics showed the pipeline was ~1.1× human (Programmer 0.6× = slower), 100+ reworks, 2d4h Director-wait — over-engineered for delivered value, especially as models strengthen.

## Decision log

### D1 — What STAYS (only the build engine changes)
Project creation, the version/work hierarchy, and metrics **stay**. More broadly, **only the build ENGINE (the agent pipeline) changes** — the surrounding infrastructure stays and adapts: cockpit, KB integration, UAT provisioning, deploy machinery, multi-module, settings, auth. (Each confirmed part-by-part below.)

### D2 — The 5 agent roles → Lead Engineer + Auditor
The 5-role serial structure (Designer→Customer→Implementer→Auditor→Coordinator, with handoffs) is **removed**. It collapses into:
- **Lead Engineer (Hlavný vývojár)** — one strong senior agent that does design + implementation + self-verification with full context, like Dedo. It does the core/hard work itself and **dynamically spawns ephemeral helper agents** for parallel/bulk sub-tasks, then integrates the results. It replaces the WHOLE 5-member team — it is NOT a renamed Coordinator. The Coordinator was a *dispatcher* (routed work between fixed roles, carried the "papers"); the Lead Engineer DOES the work and only pulls in *temporary* helpers on demand. Only the Coordinator's *Director-facing* part survives (reporting status + asking for approval), now done by the Lead Engineer itself.
- **Auditor** — a SEPARATE, independent agent that verifies the Lead Engineer's work before release (the verifier). Invoked at high-value points only: the release-acceptance check (run the app, confirm it does what the spec promised) + adversarial spot-checks on risky parts (security, money, core contract). **Stays independent from the Lead Engineer** (it is NOT one of its helpers — it checks from outside) because no agent can fully audit itself (blind-spot safeguard). See D5 for the targeted-vs-old-per-task distinction.

Note: "agents" do not disappear — the 5 standing ROLES with serial handoffs do. v2.0.0 has the Lead Engineer (the doer, with on-demand helpers) + the independent Auditor (the checker).

### D3 — Pipeline phases stay VISIBLE (transparency preserved)
**DECIDED (Director): consolidated 6-phase map — Príprava → Návrh → Plán → Programovanie → Verifikácia → Vydanie** (+ Hotovo, terminal). "Návrh" consolidates scope/API/BE/FE design; "Verifikácia" is the **Auditor's** phase (consolidates the old "Kontrola zákazníkom" + "Audit"). Sufficient transparency without faking serial sub-phases. (The Lead Engineer MAY report finer live detail within a phase if useful, but the visible map is these 6.)
- **Meaning shifts:** the display shows "which stage the **Lead Engineer** is in" (its progress), not "which agent is active."
- **Approval points (NOT "gates"):** between phases the Director can approve before continuing — these are **schvaľovacie body**, driven by the **autonomy dial** (Director chooses where to approve: release-only, key decisions, or full-auto). Distinct from the Auditor (the independent verifier, not an approval stop).

### D4 — Name of the doer agent → Lead Engineer / Hlavný vývojár
**Lead Engineer** (English — code/identifiers) / **Hlavný vývojár** (Slovak — UI). Chosen because it captures a hands-on senior who does the core work AND leads a (dynamic, temporary) team — not a pure manager who only delegates (which would drift back toward the Coordinator). Rejected: Builder/Staviteľ, Maker/Tvorca, Craftsman/Majster (construction/craft metaphors, not IT), pure Team Leader/Tímlíder (leans management).

### D5 — Name of the verifier agent → Auditor
The independent verifier is named **Auditor** (works in both EN and SK — SK form *Audítor*; familiar from the old role). **IMPORTANT — same name, leaner job:** the OLD Auditor checked **every task** (per-task) — the expensive, over-engineered part we dropped. The NEW Auditor is **targeted**: the release-acceptance check at the end + adversarial spot-checks on risky parts only — NOT per-task. (This was previously called "the Gate"; "Gate" is retired to avoid clashing with the phase approval-points.) Clean terminology: **Auditor** = independent verifier; **schvaľovacie body** = the Director's approval stops (autonomy dial).

## Lead Engineer — the concept (elaborates D2)
The **Lead Engineer (Hlavný vývojár)** owns and delivers the whole build end-to-end with full context — "Dedo, productized." It is **not literally one person**: it does the core/hard work itself and **dynamically creates temporary helper agents** for sub-tasks (parallel/bulk work), divides the work, and integrates the results — exactly how Opus/ultracode operates.
- **Full context, no handoffs:** keeps one warm context the whole way (Príprava→Návrh→Plán→Programovanie + prepares Vydanie). It remembers WHY it designed something because it designed it — no re-reading, no handoff drift (vs the old role relay).
- **Dynamic, on-demand team:** helpers are **ephemeral** (spawned for a specific sub-task, return a result, disappear) — NOT standing roles. Small task → no helpers (does it alone); large task → spawns and directs them. Scales to need.
- **Does:** preparation, design, task-plan, coding (with continuous self-checking), and directing/integrating its helpers.
- **Does NOT:** the final independent verification — that is the **Auditor** (separate agent, OUTSIDE the Lead Engineer's team), because no agent can fully audit itself. The Lead Engineer is not its own judge.
- **Not a script:** an intelligent agent that decides its own steps, within the phases + autonomy-dial approval points.

## Part-by-part decisions (filled through the consultation)
- **Project creation** — KEEP (D1). Details: TBD.
- **Version / work hierarchy** — KEEP (D1). **Customer-spec entry (Director, 2026-06-25):** the customer specification (`customer-requirements.md`, per-version at `docs/specs/versions/v<N>/customer-requirements.md` — the build's brief, read first by the Lead Engineer in Príprava, today read by the pipeline) is entered **directly in the New Version flow** (a "Zákaznícka špecifikácia" editor), NOT via the deep file-browser path (current pain: Špecifikácie → Projects → Docs → Specs → Version → save file = unintuitive). On save it writes to the per-version path automatically. Editable later from the version detail. The Špecifikácie browser still shows/edits the same file (added entry point, not a replacement). Required before a build can start (it is the brief).
- **Metrics & ROI** — KEEP feature; **computation basis per-AGENT → per-PHASE** (Príprava→Návrh→Plán→Programovanie→Verifikácia→Vydanie). All sources attribute to the phase they ran in (Lead Engineer → current phase; helpers → spawning phase; Auditor → Verifikácia). Per-role rates/wages in Settings → per-phase. Agent-vs-human comparison stays, per phase. Historical v1.0.0 data is per-role (not 1:1 comparable). [Director, 2026-06-25]
- **Orchestrácia screen** (the /cockpit board for a selected version) — see D2/D3. **Layout (Director, 2026-06-25):**
  - **Pipeline = HORIZONTAL phase bar at the TOP** (like NEX Command — NOT a second left rail; saves left space): `Príprava ✓ › Návrh ✓ › Plán ● › Programovanie ○ › Verifikácia ○ › Vydanie ○` (✓ done · ● current · ○ pending), each phase **clickable**.
  - **Each phase is its own TAB with PERMANENT content** — clicking a phase shows that phase's artifacts + activity, and they **persist after the build completes** (FIXES the old pain: the task plan was only visible during the build, then vanished irretrievably). Tab contents: **Príprava** (intake/goal), **Návrh** (design/specs API+BE+FE), **Plán** (task plan ← used to vanish), **Programovanie** (coding log + task progress), **Verifikácia** (Auditor verdict + findings), **Vydanie** (release info). Current phase = live; finished phases = durable record. **The phase chips in the top bar ARE the tabs** — there is NO separate tab row (saves space). Two states coexist on the bar: **● = where the build currently is** (auto-advances); **highlighted = which tab the Director is viewing** (their click). They can differ — e.g. review the finished "Plán" tab while the build runs ahead in "Programovanie."
  - **Requires durable persistence (not just UI):** each phase stores its output as an artifact/record (design, plan, verdict), not in-memory only → also serves as the build's audit trail.
  - Below the tabs: who's-up status (Lead Engineer / + helpers / Auditor / awaiting Director), action buttons at **schvaľovacie body** (Schváliť / Uprav / Pokračovať) + Spustiť, and a drawer to peek the raw Lead Engineer terminal.
  - Simplified vs today (one Lead-Engineer line + Auditor, instead of 5-role ping-pong). **Orchestrácia = manager view (overview + approvals); Hlavný vývojár = the raw terminal (watch/talk live).** Version-scoped.
  - **Plán + Programovanie tab — UX requirements (Director, 2026-06-25):**
    1. **Remember the plan-tree expand/collapse state** (EPIC › FEAT › TASK). Navigating away (e.g. to Metriky) and back must NOT reset it — today it does (a bug). Persist across navigation; ideally also across page reload (per user).
    2. **Color-code the levels** for readability in large plans: **EPIC = purple, FEAT = yellow, TASK = blue.** (Verify legibility — esp. yellow on the light theme; shade may be tuned — "uvidíme v reáli".)
    3. **Programovanie tab = split view:** live programming activity on the LEFT + the **task plan on the RIGHT** (watch both at once). Keep this.
- **Autonomy dial** (SK UI name: proposed **"Miera autonómie"** — pending Director) — a Director setting controlling how much the Lead Engineer does on its own vs how often it stops at a **schvaľovací bod** for approval. Preset levels: **Plná autonómia** (whole build, reports only at the end / if truly stuck) · **Len pri vydaní** (autonomous through design+build, stops only before Vydanie) · **Pri kľúčových bodoch** (stops e.g. after Návrh + before Vydanie) · **Po každej fáze** (stops after each phase — max control, like the old way). Global default + overridable per project / per build (sensitive → more stops; routine → full-auto). The schvaľovacie body are the potential stops; the dial decides which actually halt. **Fast-fix = dial at full-auto.** Solves the old forced-gate problem (the 2d4h Director-wait). [Director, 2026-06-25]
- **Verification — the Auditor (TWO touchpoints, Director 2026-06-25):**
  - **(a) Upfront spec/design review** (after Návrh, before the build commits to coding) — the Auditor independently scans the brief (`customer-requirements.md`) + the Lead Engineer's design for holes / ambiguities / contradictions = **the OLD Customer agent's Gate-E function, now done by the independent Auditor**. PLUS the Lead Engineer raises its own clarification questions during Príprava/Návrh (it can't build what isn't specified). Both surface at the **schvaľovací bod after Návrh**: the Director fixes/clarifies the brief or approves to proceed. (Replaces the old /dialogue page — inline in the flow now.)
  - **(b) End verification** at the Verifikácia phase — release-acceptance (run the app, confirm it does what the brief promised) + adversarial spot-checks on risky parts. NOT per-task.
  - Depth is **dial-able** (autonomy dial): full independent upfront review for important/regulated projects; lighter (just the Lead Engineer's questions) for quick ones.
  - Net: the **Auditor = independent conscience at both ends** — upfront "is the plan sound?" + end "did it deliver?". (Updates charter §2.5 — the upfront pillar is now the Auditor's early review, not a standing Customer role.)
- **Fast-fix lane ("Rýchla oprava") — KEEP, simplified (Director, 2026-06-25).** For small fixes that don't warrant a full version build (no customer-spec, no 6 phases). Flow: describe the fix (a **directive**) → **auto-create a patch version** (vX.Y.Z+1) → the **Lead Engineer** takes the directive (which IS the brief) and fixes it directly on a **short path** (skips the heavy Návrh/Plán) → **light Auditor check** (fix works + no regression — focused, NOT the full release oracle) → **auto-deploy** → **autonomous** (zero mid-flight approvals by default; dial-able to require approval for sensitive fixes). Outside the 6-phase waterfall — its own minimal lane (Oprava → quick verify → Deploy). Fits v2.0.0 better than today (one Lead Engineer instead of 5-role routing). Proven today: nex-ledger fast-fix ~18 min, zero approvals.
