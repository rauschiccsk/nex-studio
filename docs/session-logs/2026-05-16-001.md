## Key decisions

- **Customer agent ako 4. plnohodnotný ICC agent** — nie len ad-hoc role v dialogue UI. Plný charter (~270 LOC) s explicit identity ("ľudský zákazník, nie technický"), tool allow-list (Read all + Write iba `customer-dialogue.md` + state + session-logs), 7-batch coverage workflow (auth → happy path → modules → screens → errors → edge cases → integrations), TODO logging triáž za Designer reply, anti-pattern list ("nepýtať sa o Pydantic / SQL / asyncio"). Kodifikácia pattern ktorý Director discovered organically, teraz reproducible cross-project.

- **Vlastná PTY namespace pre dialogue** (nie reuse agent_terminal) — agent_terminal enforce single-session-per-role constraint cez DB partial unique index, ktorý by koliziovalo s dialogue's Designer + prípadný standalone Designer terminál v `/designer`. Trade-off: copy-paste low-level PTY helpers z agent_terminal namiesto extract spoločného modulu. Decision: keep copy-paste pre v1, extract môže byť future refactor keď oba shapes stabilizujú (agent_terminal je live-terminal use case, dialogue je headless-orchestration use case — invariants sú subtly odlišné).

- **Settle-and-flush pattern pre PTY output collection** — agenti píšu odpovede v token-streamed chunkoch. Bez settle window by sa každý chunk persistoval ako samostatná DialogueMessage row → fragmented konverzácia. Riešenie: collector bytearray + restart-able settle task; flush keď je buffer 2.5s ticho. Trade-off: pri pomalom claude tokenize (LLM thinking) môže settle prematurely fire → "split" message. Worth sledovať pri prvom live run.

- **Plný-gate mode default pre v1** — alternatíva (auto-flow s pause) by bola rýchlejšia ale Director's "chcem byť presvedčený o kvalite" argument vyhráva. Plný-gate kvalita > rýchlosť, prvá iteration je Director's learning of the pattern. Per-project config-driven prepínanie môže byť v0.2.0.

- **Gate E pipeline placement** — medzi Gate D (FE trio complete) a Implementer handoff. Conceptuálne legitimate: Designer skončil syntézu, spec je technically complete, ale "robí to čo zákazník čaká?" ešte nebola validovaná. Plus mandatory pre regulated-ledger + regulated-payroll (compliance-heavy projekty kde missed edge case = audit fail), optional inde. Codifies CR-001 pattern z NEX Inbox (Tibor + Director's pre-Implementer expert review) ako structured agent-driven process.

- **`{{PROJECT_SLUG}}` substitution v deny rules** — Customer settings.json template má `Write(/opt/projects/{{PROJECT_SLUG}}/backend/**)` v deny (per L-016 lesson: absolute paths in deny, no relative globs). init.sh `substitute()` ho prepíše per projekt pri bootstrap-e. Konzistentné so všetkými 3 ostatnými agentmi.

- **Manuálny bootstrap nex-inbox** — init.sh nebehne pre existing project (would override existing files). Pre Customer agent integration do nex-inbox použitý sed-based template substitution. Pre budúce projekty: init.sh automaticky vytvorí všetkých 4 agentov.
